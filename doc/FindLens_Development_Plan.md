# FindLens 在线镜头选型与 3D 视场模拟服务开发计划

## 1. 项目目标

FindLens 面向需要选择相机镜头的用户，提供从光学参数计算到直观场景理解的一体化服务。

用户既可以根据传感器与镜头参数计算视场，也可以从实际需求出发，例如“在 5 米处完整拍到一个成年人”，由系统反算建议焦距，并通过 3D 场景验证结果。

项目采用分阶段建设策略：

- Plan 1 先完成最简单但功能闭环的版本，验证计算逻辑、3D 体验与选型流程。
- Plan 2 开始扩充常用对象、场景与可比较能力。
- Plan 3 引入真实镜头/相机数据及专业判断能力。
- Plan 4 面向公网服务、用户协作和商业化运营。

## 2. 产品边界与术语

### 2.1 计算对象

系统应明确区分：

| 名称 | 含义 | 示例 |
| --- | --- | --- |
| 视场角（Angle of View） | 镜头可覆盖的角度范围 | 水平 29.6 deg |
| 物方视野（Field of View） | 指定距离处可覆盖的实际宽高 | 5 m 处覆盖 2.64 m x 2.21 m |
| 画面占比 | 目标物体在成像画面宽度/高度中所占比例 | 人物高度占画面 79% |
| 像素覆盖 | 目标在成像图像中占用的像素数量 | 人物高度约 1620 px |

### 2.2 传感器参数规则

计算内部以真实尺寸为准：

- 传感器宽度，单位 `mm`
- 传感器高度，单位 `mm`
- 水平/垂直分辨率，单位 `px`
- 像元尺寸，单位 `um`

输入分辨率和像元尺寸时，自动计算传感器尺寸：

```text
传感器宽度 mm = 水平像素数 x 像元尺寸 um / 1000
传感器高度 mm = 垂直像素数 x 像元尺寸 um / 1000
```

“1/2 英寸”“1/1.8 英寸”等 optical format 仅作为用户可识别的规格标签，不能直接替代真实宽高参与精确计算。

### 2.3 基础光学模型

第一阶段采用理想几何模型，目标是帮助选型与理解覆盖范围，不模拟真实镜头成像质量。

```text
水平视场角 = 2 x atan(传感器宽度 / (2 x 焦距))
垂直视场角 = 2 x atan(传感器高度 / (2 x 焦距))

距离处水平覆盖范围 = 2 x 距离 x tan(水平视场角 / 2)
距离处垂直覆盖范围 = 2 x 距离 x tan(垂直视场角 / 2)

由目标水平视野反算焦距 = 距离 x 传感器宽度 / 目标水平视野
由目标垂直视野反算焦距 = 距离 x 传感器高度 / 目标垂直视野
```

后续可加入畸变、景深、解析力、真实镜头参数等专业能力。

## 3. 总体用户路径

### 3.1 从参数查看效果

```text
选择/输入传感器参数
  -> 输入镜头焦距
  -> 查看视场角
  -> 放置目标物体并设置距离
  -> 查看实际覆盖范围、画面占比和 3D 效果
  -> 切换标准焦距比较差异
```

### 3.2 从需求反推镜头

```text
选择/输入传感器参数
  -> 选择目标物体与距离
  -> 设定完整入镜或期望画面占比
  -> 系统反算理论焦距
  -> 推荐最接近的标准焦距
  -> 在 3D 场景与镜头视角中确认效果
```

## 4. 总体架构方向

初期以本地部署为主，技术架构应支持后续平滑上线，而不过早引入账号、权限和复杂数据库。

### 4.1 推荐技术构成

| 模块 | 建议技术 | 说明 |
| --- | --- | --- |
| Web 应用 | Nuxt + Vue + TypeScript | 页面、交互和后续服务端能力统一管理 |
| 3D 展示 | Three.js，配合 Vue 组件封装 | 展示相机、视锥、人物和距离关系 |
| 计算核心 | 独立 TypeScript 模块 | 公式集中管理，可被页面和后端复用 |
| 数据存储（初期） | JSON 配置文件 | 只维护少量预设，启动成本低 |
| 数据存储（后期） | PostgreSQL | 镜头库、相机库、用户方案等结构化数据 |
| 本地部署（Plan 1） | Orin Ubuntu + Node.js LTS + systemd | 直接运行应用，降低验证阶段部署复杂度 |
| 公网部署（后期） | Docker + 反向代理 + HTTPS | 部署到 x86 Ubuntu 云服务器时启用 |

### 4.2 模块划分

| 模块 | 责任 |
| --- | --- |
| 参数输入模块 | 传感器、焦距、距离、目标尺寸输入及合法性校验 |
| 光学计算模块 | 正向计算、反向计算、画面占比、像素覆盖 |
| 3D 场景模块 | 视锥、人物、地面标尺、拖动和镜头画面视角 |
| 推荐模块 | 将理论焦距匹配为标准焦距或真实镜头型号 |
| 数据模块 | 传感器预设、镜头数据、物体模板与后续用户方案 |
| 分享/输出模块 | 参数化 URL、截图或报告导出 |

### 4.3 部署平台与兼容约束

Plan 1 的本地部署目标设备为 NVIDIA Jetson AGX Orin，其操作系统为 Ubuntu，CPU 架构为 `linux/arm64`。未来公网部署预计采用常见 Ubuntu x86 服务器，CPU 架构为 `linux/amd64`。

两种平台不会改变产品功能设计，但开发阶段必须遵守以下约束，以避免后续迁移时重写应用：

- 主应用仅依赖标准 Node.js Web 运行环境，不依赖 Orin 的 CUDA、TensorRT 或 Jetson 专用驱动。
- 3D 场景由访问页面的用户浏览器通过 Three.js 渲染，Orin 仅负责提供 Web 应用和数据。
- 避免引入只支持单一 CPU 架构的 Node.js 原生扩展或二进制工具；确需引入时，必须核对 `arm64` 与 `amd64` 均可运行。
- 配置使用环境变量或独立配置文件管理，不在业务代码中写死 Orin 的目录、网络地址或系统环境。
- 后续开始容器化时，Docker 镜像应支持 `linux/arm64` 与 `linux/amd64`，使本地设备和公网服务器均可复用发布流程。

## Plan 1：最小完整版本（MVP）

### 目标

用极少的选项实现完整流程：既能计算，也能反算；既能看到数值，也能通过一个可拖动的成年人模型直观判断镜头效果。

Plan 1 的重点不是建立大而全的数据仓库，而是验证以下问题：

- 用户能否顺利理解传感器、焦距、距离与拍摄范围的关系。
- 3D 人物展示是否显著提升选型直觉。
- 从实际拍摄目标反算焦距的流程是否好用。

### 功能范围

#### 1. 传感器输入

保留两种输入方式：

- `手动输入`：传感器宽度和高度。
- `按像元计算`：水平/垂直分辨率 + 像元尺寸，自动换算传感器宽高。

仅提供 2 至 3 个常见传感器预设作为示例，不建设完整传感器库。

#### 2. 正向镜头计算

用户输入焦距后，实时显示：

- 水平、垂直、对角视场角。
- 当前人物距离处的水平、垂直覆盖范围。
- 人物在画面中的宽度占比和高度占比。
- 如已输入分辨率，显示人物所覆盖的像素宽度和高度。

#### 3. 反向焦距计算

Plan 1 支持两种反算方式：

- 指定距离处，要求成年人完整入镜，系统计算可用焦距上限或建议焦距。
- 指定距离处，设置人物期望画面高度占比，例如 `70%`，系统计算理论焦距。

输出：

- 理论焦距。
- 从少量标准焦距中选择的推荐结果：`6 mm`、`8 mm`、`12 mm`、`16 mm`、`25 mm`、`35 mm`、`50 mm`。
- 采用推荐焦距后的实际覆盖范围和人物画面占比。

#### 4. 3D 视场展示

只提供一个目标对象：`成年人`。

默认人物尺寸：

```text
身高：1.70 m
宽度：0.45 m
厚度：0.25 m
```

用户可修改身高、宽度和厚度，不加入儿童、车辆、树木等预设选项。

3D 场景包含：

- 一个固定位置的相机/镜头示意模型。
- 从相机向前展开的半透明视锥体。
- 一个可识别但轻量的人物模型。
- 地面距离网格与标尺。
- 人物所在距离位置的视场截面框。

#### 5. 3D 交互

第一版交互控制保持简单：

- 拖动人物前后移动，修改距离。
- 用输入框或滑块精确设置人物距离。
- 修改焦距时，视锥与镜头预览实时更新。
- 修改人物尺寸时，画面占比实时更新。
- 场景支持基础旋转、缩放和复位，不提供复杂布置编辑。

Plan 1 不要求人物自由横向摆放；人物默认位于光轴中央，以优先验证“距离和构图比例”这一核心需求。

#### 6. 两种观察视图

必须提供：

- `空间视图`：从斜上方看到相机、视锥、地面距离与人物。
- `镜头画面预览`：从相机视角看到人物在最终画面中的占比。

两者应同步更新，用户拖动人物或切换焦距后不需要手动刷新。

#### 7. 基础推荐结果

Plan 1 的推荐对象是“标准焦距”，不是厂商真实 SKU。

展示方式示例：

```text
理论焦距：14.7 mm
推荐标准焦距：16 mm
5 m 处覆盖范围：2.64 m x 2.21 m
人物高度占画面：76.9%
状态：人物可完整入镜
```

#### 8. 场景分享

支持将当前核心参数编码到 URL，便于复现场景：

- 传感器宽高。
- 分辨率及像元尺寸（若已填写）。
- 焦距。
- 人物尺寸。
- 人物距离。
- 当前模式（正向/反向）。

#### 9. Orin 本地直接部署

Plan 1 在 Jetson AGX Orin 的 Ubuntu 系统上直接部署，不要求 Docker。

推荐运行结构：

```text
Jetson AGX Orin / Ubuntu ARM64
  -> Node.js LTS
  -> FindLens Nuxt 应用
  -> JSON 配置数据
  -> systemd 进程管理与开机自启
```

局域网用户可通过 Orin 的 IP 地址访问应用。Plan 1 不要求域名、HTTPS、反向代理、PostgreSQL 或 GPU 运行环境配置。

### Plan 1 页面设计建议

Plan 1 控制页面数量，只提供一个主工作台页面。

```text
+-----------------------------------------------------------+
| FindLens: 镜头选型与视场模拟                              |
+-------------------------+---------------------------------+
| 参数与计算面板          | 3D 空间视图                    |
| - 传感器                | - 相机与视锥                   |
| - 焦距/反算模式         | - 可前后拖动的人物             |
| - 人物距离与尺寸        | - 距离标尺与视场截面           |
| - 数值结果              |                                 |
+-------------------------+---------------------------------+
| 标准焦距推荐            | 镜头画面预览                  |
+-------------------------+---------------------------------+
```

### Plan 1 非目标

以下内容明确不进入 Plan 1，避免范围失控：

- 车辆、树木、纸箱等更多物体预设。
- 多人或多物体场景。
- 真实厂商镜头/相机型号数据库。
- 用户登录、方案保存和后台管理。
- 畸变、景深、光圈、对焦质量模拟。
- PDF 报告或询价流程。
- 移动端精细适配；可保证基本可打开，但以桌面浏览器为主。

### Plan 1 验收标准

| 类别 | 验收标准 |
| --- | --- |
| 参数 | 用户可输入传感器尺寸，或由分辨率与像元尺寸推导传感器尺寸 |
| 正向计算 | 输入焦距和人物距离后，能实时显示视场角、覆盖范围、人物占比 |
| 反向计算 | 给定人物距离与期望占比或完整入镜目标，能反算理论焦距 |
| 推荐 | 能从固定标准焦距列表给出推荐，并显示使用后的结果差异 |
| 3D 展示 | 可看到相机、视锥、人物、距离标尺与当前距离截面 |
| 操作 | 用户能通过拖动或滑块改变人物距离，数值与两个视图同步变化 |
| 画面预览 | 能从镜头第一视角显示人物是否完整入镜及其画面占比 |
| 分享 | 刷新或分享包含参数的 URL 后可还原当前核心场景 |
| 测试 | 核心光学计算与反向计算具备自动化测试，边界输入有提示 |
| 部署 | 项目可在 Jetson AGX Orin 的 Ubuntu ARM64 环境直接运行，使用 Node.js LTS 与 systemd 管理，不要求 Docker |
| 迁移兼容 | 代码与依赖保持兼容未来 Ubuntu x86_64 服务器部署，不耦合 Jetson 专属运行能力 |

### Plan 1 建议开发步骤

1. 固化单位、字段和计算公式，并准备计算测试用例。
2. 完成单页面参数区、正向计算与反向计算。
3. 建立标准焦距推荐逻辑与结果展示。
4. 建立最简 3D 场景：相机、视锥、地面、人物和距离截面。
5. 打通人物拖动、参数面板与镜头画面预览联动。
6. 增加 URL 场景分享与输入边界校验。
7. 整理 Orin Ubuntu 上的 Node.js 安装、应用启动与 systemd 服务配置，完成 MVP 验收。

## Plan 2：常用对象与场景比较版本

### 目标

在 Plan 1 的核心体验得到验证后，将“只能看一个人”升级为覆盖日常选型场景的实用工具，并增强比较决策能力。

### 主要功能

- 增加物体模板：儿童、轿车、SUV、卡车、树、门、纸箱、托盘。
- 支持用户创建一个自定义长宽高目标物体。
- 支持人物/车辆横向移动和车辆朝向旋转。
- 支持同一场景放置少量多个对象，如两个人或一人一车。
- 增加俯视平面图，特别服务于道路、门口、停车位等横向覆盖问题。
- 提供多个焦距并排比较，例如 `12 mm / 16 mm / 25 mm` 的画面预览。
- 增加典型场景模板：
  - 室内门口看人。
  - 道路看车。
  - 仓储区看托盘或纸箱。
  - 户外区域覆盖。
- 支持场景截图导出。

### 数据与交互升级

- 对象预设转为结构化配置数据。
- 保存最近使用的参数到浏览器本地存储。
- URL 支持恢复多个物体和当前视图状态。
- 增加清晰的对象越界、被裁切和留边过多提示。

### 非目标

- 不强制引入真实镜头型号数据库。
- 不做账号系统和云端方案管理。
- 不做严格成像质量或识别能力保证。

### 验收重点

- 用户能通过常见物体模板快速建立实际应用场景。
- 同一需求下可以直观看出不同标准焦距的构图差异。
- 多物体和自定义尺寸不会破坏计算结果与交互一致性。

### 部署策略

Plan 2 可继续沿用 Orin 上的直接部署方式。若局域网使用频率增加，可按需加入反向代理或更规范的服务日志管理，但容器化仍不是本阶段强制要求。

## Plan 3：真实选型与工程辅助版本

### 目标

将工具从视觉化计算器升级为可以帮助工程师挑选实际镜头和相机组合的选型平台。

### 主要功能

- 建立真实传感器/相机型号库。
- 建立真实镜头型号库，维护以下字段：
  - 品牌、型号、焦距、接口类型。
  - 最大支持成像面或靶面尺寸。
  - 光圈范围、最近工作距离。
  - 可获取的畸变、分辨率或产品资料链接。
- 根据计算需求筛选真实可用型号。
- 增加接口匹配、成像面覆盖、最近工作距离等硬性规则校验。
- 加入像素覆盖判断：
  - 人体或车辆在图像中覆盖多少像素。
  - 自定义关注区域在图像中覆盖多少像素。
- 支持导出选型方案摘要，包含参数、场景截图和候选型号。
- 增加数据维护页面或受限后台导入工具，支持 CSV/Excel 数据导入。

### 专业模拟增强

按真实数据可获取程度逐步加入：

- 镜头畸变的近似可视化。
- 对焦距离和最近工作距离提示。
- 简化景深估算。
- 目标像素覆盖的工程阈值提示。

所有模拟或阈值判断均应标明依据和误差边界，不将近似计算包装为实测结果。

### 验收重点

- 根据一个实际目标场景，系统可以返回满足硬性约束的真实镜头候选列表。
- 每个推荐结果可以说明为何推荐、为何淘汰其他镜头。
- 导出的结果足以支持内部方案讨论或初步客户沟通。

### 部署策略

若 Plan 3 引入 PostgreSQL、后台管理并进入持续内部使用，可根据维护成本决定是否提前引入 Docker Compose；该决定应服务于数据库备份、升级和部署一致性，而不是 Plan 1 的前置条件。

## Plan 4：公网服务与运营版本

### 目标

将已验证的本地/内部产品部署为可公开访问、可维护和可逐步运营的在线服务。

### 主要功能

- 面向 Ubuntu x86_64 公网服务器进行 Docker 容器化部署，配置域名与 HTTPS。
- 数据库存储传感器、镜头、对象模板、用户方案和版本信息。
- 管理员登录、角色权限和数据维护流程。
- 用户账户与方案保存，可生成分享链接或只读方案页面。
- 生成带品牌信息的选型报告或询价信息。
- 基础访问统计、错误监控、日志与备份恢复。
- API 限流、登录保护、内容安全与隐私合规处理。

### 商业能力候选

- 厂商或代理商镜头库展示。
- 推荐镜头跳转到产品页或询价页。
- 项目方案协作与客户留资。
- 公开计算器免费使用，高级选型报告或数据能力作为扩展服务。

### 验收重点

- 公网稳定访问且 HTTPS、备份、监控和权限机制完备。
- 公开用户能够完成计算、3D 查看、实际型号选择和结果分享。

## 6. 当前需要优先确认的产品决策

在进入 Plan 1 详细交互原型或开发前，建议确认以下事项：

1. Plan 1 的首要用户是镜头销售/方案人员，还是最终客户自行试算；这决定界面措辞和默认参数复杂度。
2. 反向计算默认目标应采用“人物完整入镜”还是“人物高度占画面指定比例”；两者都可提供，但默认入口只能突出一个。
3. 人物距离的有效范围，例如 `0.5 m - 50 m`，以及是否需要常用距离快速按钮。
4. Plan 1 是否在桌面浏览器上验证即可，还是从第一天就要求手机端拖动可用。
5. Plan 3 的真实镜头数据来源将由自有产品资料维护，还是需要兼容多个厂商公开参数。

## 7. 风险与控制原则

| 风险 | 影响 | 控制方式 |
| --- | --- | --- |
| 用户将 3D 展示理解为真实成像结果 | 产生错误选型预期 | 明确显示其为几何覆盖模拟，后续模拟能力单独标注 |
| 第一阶段加入过多物体和数据 | 开发慢、交互混乱 | Plan 1 只保留成年人和标准焦距 |
| 真实镜头字段不完整或规格不一致 | 推荐难以可信 | Plan 3 前先定义数据规范与数据来源 |
| 3D 性能影响使用体验 | 页面加载慢或操作卡顿 | 采用轻量模型，优先桌面端，控制场景复杂度 |
| 计算单位和输入方式混淆 | 结果错误 | 全部数值标明单位，核心公式自动化测试 |
| Orin ARM64 与未来公网 x86_64 存在架构差异 | 依赖或部署方式迁移失败 | 主应用保持平台中立，避免 Jetson 专属耦合，容器化阶段采用双架构支持 |
